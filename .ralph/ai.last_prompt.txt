# Jordan HEDGE

**Personal reasoning engine for Jordan DeHerrera.**  
A HEDGE-inspired (Hypothesis-Evidence-Driven-Graph-Evaluation) system for maintaining calibrated belief states across life, work, and project threads.

---

## What This Is

Most productivity tools track tasks. This tracks *beliefs about what matters* â€” with explicit uncertainty.

Inspired by MEL's clinical HEDGE engine, adapted for personal reasoning:
- **Threads** replace patients (estate, malpractice, MEL, health, work)
- **Signals** replace symptoms (deadline proximity, unanswered emails, task staleness)
- **Hypothesis types** replace diagnoses (URGENT_ACTION_NEEDED, AT_RISK, BINDING_CONSTRAINT)
- **Relations** encode how signals update beliefs (LLR_pos, LLR_neg)

Multiple weak signals compound. A thread with a nearby deadline + unanswered email + financial exposure gets a dramatically higher AT_RISK posterior than any single signal alone. That's the Bayesian advantage over a task list.

---

## Architecture

```
jordan-hedge/
  db/schema.sql          â€” SQLite schema (concepts, relations, facts, hypotheses, threads, actions)
  ontology/seed.json     â€” Concepts, relations, initial threads
  src/engine/
    db.py                â€” Connection + custom SQLite functions (logit_to_prob, belief_entropy, etc.)
    hedge_engine.py      â€” Core engine: ingest signals, update hypotheses, generate actions
  scripts/
    seed.py              â€” Initialize database from ontology/seed.json
    status.py            â€” Print current belief state dashboard
```

### Core Loop

1. **Signal ingestion** â†’ facts inserted into SQLite
2. **SQLite trigger fires** â†’ updates log_odds_posterior for all hypotheses on that thread
3. **Probability = sigmoid(log_odds_posterior)** via custom SQLite function
4. **Actions generated** â†’ ordered by `expected_utility = P Ã— risk_weight Ã— (1 - uncertainty)`
5. **Status dashboard** â†’ surfaces what to do right now
